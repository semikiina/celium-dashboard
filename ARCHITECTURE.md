# Architecture ‚Äî Celium Dashboard

## Overview

The dashboard follows a straightforward **server-first Next.js App Router** pattern. Pages fetch data server-side where possible; client components are used only when interactivity or browser APIs are required (charts, map, real-time polling).

```
Browser
  ‚îî‚îÄ‚îÄ Next.js App Router
        ‚îú‚îÄ‚îÄ Server Components  ‚Üí  fetch from Supabase directly (no extra API hop)
        ‚îú‚îÄ‚îÄ Client Components  ‚Üí  SWR hooks  ‚Üí  Next.js API Routes  ‚Üí  Supabase
        ‚îî‚îÄ‚îÄ API Routes (/api)  ‚Üí  Supabase JS client
```

## Database Design (Supabase / Postgres)

### Table: `nodes`

Represents a single hardware node in the network.

```sql
CREATE TABLE nodes (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  external_id   INTEGER UNIQUE NOT NULL,       -- hardware-assigned node ID
  name          TEXT NOT NULL,                 -- human-readable label e.g. "Node-042"
  type          TEXT NOT NULL CHECK (type IN ('gateway', 'relay', 'end_node')),
                                   -- gateway = C3, relay = C2, end_node = C1
  status        TEXT NOT NULL DEFAULT 'unknown'
                  CHECK (status IN ('online', 'offline', 'warning', 'unknown')),
  gateway_id    UUID REFERENCES nodes(id),     -- NULL for C3 gateways; points to parent C3 or C2 for end_nodes and relays
  lat           NUMERIC(10, 7),                -- GPS latitude
  lng           NUMERIC(10, 7),                -- GPS longitude
  firmware_ver  TEXT,
  hardware_ver  TEXT,
  deployed_at   TIMESTAMPTZ,
  last_seen_at  TIMESTAMPTZ,
  created_at    TIMESTAMPTZ DEFAULT now()
);
```

### Table: `readings`

Time-series sensor telemetry from nodes. One row per transmission event.

```sql
CREATE TABLE readings (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  node_id         UUID NOT NULL REFERENCES nodes(id) ON DELETE CASCADE,
  timestamp       TIMESTAMPTZ NOT NULL DEFAULT now(),

  -- Environmental sensors (nullable ‚Äî not all nodes carry all sensors)
  temperature     NUMERIC(5, 2),               -- ¬∞C
  humidity        NUMERIC(5, 2),               -- %RH
  pressure        NUMERIC(7, 2),               -- hPa

  -- Radio / link metrics
  rssi            INTEGER,                     -- dBm, received signal strength
  snr             NUMERIC(5, 2),               -- dB, signal-to-noise ratio
  spreading_factor INTEGER,                    -- LoRa SF7‚ÄìSF12

  -- Power
  battery_voltage NUMERIC(4, 3),              -- V, e.g. 3.721
  battery_pct     INTEGER,                     -- 0‚Äì100

  -- Network
  hop_count       INTEGER,                     -- number of hops to gateway
  seq_num         INTEGER,                     -- packet sequence number (for loss detection)

  raw_payload     JSONB                        -- full raw JSON from node (extensible)
);

-- Index for time-series queries
CREATE INDEX readings_node_time_idx ON readings (node_id, timestamp DESC);
```

### Table: `alerts`

Generated alerts ‚Äî for MVP these can be seeded or auto-generated by a Postgres function/trigger.

```sql
CREATE TABLE alerts (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  node_id     UUID REFERENCES nodes(id) ON DELETE SET NULL,
  severity    TEXT NOT NULL CHECK (severity IN ('info', 'warning', 'critical')),
  type        TEXT NOT NULL,                   -- e.g. 'node_offline', 'low_battery', 'high_temp'
  message     TEXT NOT NULL,
  resolved    BOOLEAN DEFAULT false,
  created_at  TIMESTAMPTZ DEFAULT now(),
  resolved_at TIMESTAMPTZ
);
```

### Typography

All text uses **Inter**, loaded via `next/font/google` in `app/layout.tsx` as **`--font-inter`** (avoid naming it `--font-sans`, which clashes with Tailwind‚Äôs default). `app/globals.css` maps `font-sans`, `font-heading`, and `font-body` to `var(--font-inter)`.

| Role | Weight | Tailwind class pattern |
|---|---|---|
| Page titles, section headings | 700 (bold) | `font-heading font-bold` |
| Card titles, labels | 600 (semi-bold) | `font-heading font-semibold` |
| Body text, descriptions | 400 (regular) | `font-body` |
| UI labels, badges, metadata | 500 (medium) | `font-body font-medium` |

### Logo Usage

All logo SVGs live in `/branding/logos/`. Import them as React components using Next.js SVG import or as `<Image>` from `next/image`.

| File | When to use |
|---|---|
| `logo-full.svg` | Expanded sidebar, splash/onboarding screens |

Never use a raster version of the logo. Never recolour SVGs inline ‚Äî use the correct variant file instead.

## Data Fetching Pattern

- **Server Components** (static or infrequently updated data): call Supabase directly using the server client. No API route needed.
- **Client Components** (charts, map, explorer): use SWR hooks that call Next.js API routes. API routes use the Supabase server client.
- **Polling**: SWR `refreshInterval` uses `REFRESH_INTERVAL` (30s) for alerts and network stats. Hooks that surface latest telemetry (`useNodes`, `useNode`, `useReadings`, and `/api/nodes` inside `useOverviewData`) use `READINGS_REFRESH_INTERVAL` (5s) so the UI tracks the local readings simulator without a full reload.

## Scalability Notes (post-MVP)

These decisions keep the MVP simple without closing doors:

| Decision | How it scales |
|---|---|
| `raw_payload JSONB` column | New sensor fields are stored without schema changes |
| `readings` partitioned by time | Add Postgres time partitioning later without app changes |
| API routes as a clean boundary | Can be extracted to a separate backend service |
| SWR polling | Drop-in replacement with WebSockets or Supabase Realtime subscriptions |
| react-leaflet map | Add clustering (supercluster) and tile caching as node count grows |
| No auth now | NextAuth or Supabase Auth slots in at the middleware layer |
